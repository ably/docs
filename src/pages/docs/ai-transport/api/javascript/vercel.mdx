The Vercel integration provides a pre-built codec and three pre-bound factories for the Vercel AI SDK. It maps Vercel's `UIMessageChunk` events to `UIMessage` objects, so you can use AI Transport with `useChat` without writing a custom codec.

<Code>
```javascript
import {
  UIMessageCodec,
  createClientTransport,
  createServerTransport,
  createChatTransport,
} from '@ably/ai-transport/vercel';
```
</Code>

React-side hooks (`useChatTransport`, `useMessageSync`, `ChatTransportProvider`) live on the [React hooks reference](/docs/ai-transport/api/react/providers).

## UIMessageCodec <a id="ui-message-codec"/>

A pre-built `Codec<UIMessageChunk, UIMessage>` for the Vercel AI SDK. It handles encoding and decoding of Vercel's message types over Ably channels.

You do not need to import `UIMessageCodec` directly if you use the Vercel-pre-bound transport factories or the React hooks; they bind the codec automatically.

<Code>
```javascript
import { UIMessageCodec } from '@ably/ai-transport/vercel';
```
</Code>

## Create a Vercel client transport <a id="create-client-transport"/>

<MethodSignature>{`function createClientTransport(options: VercelClientTransportOptions): ClientTransport<UIMessageChunk, UIMessage>`}</MethodSignature>

Pre-bound version of the core `createClientTransport` with `codec: UIMessageCodec` already supplied and `api` defaulted to `/api/chat`.

<Code>
```javascript
import * as Ably from 'ably';
import { createClientTransport } from '@ably/ai-transport/vercel';

const ably = new Ably.Realtime({ authUrl: '/auth' });
const channel = ably.channels.get('conversation-42');

const transport = createClientTransport({ channel });
```
</Code>

## Create a chat transport <a id="create-chat-transport"/>

<MethodSignature>{`function createChatTransport(transport: ClientTransport<UIMessageChunk, UIMessage>, chatOptions?: ChatTransportOptions): ChatTransport`}</MethodSignature>

Create a `ChatTransport` adapter around an existing `ClientTransport`. The `ChatTransport` is the object you pass to Vercel's `useChat` as `transport`. In most cases you will use the [`ChatTransportProvider`](/docs/ai-transport/api/react/providers) and read the adapter via `useChatTransport` rather than calling this factory directly.

<Code>
```javascript
import { createClientTransport, createChatTransport } from '@ably/ai-transport/vercel';

const clientTransport = createClientTransport({ channel });
const chatTransport = createChatTransport(clientTransport);
```
</Code>

### Parameters <a id="create-chat-transport-params"/>

<Table id='CreateChatTransportParameters'>

| Parameter | Required | Description | Type |
| --- | --- | --- | --- |
| transport | Required | The underlying client transport. | `ClientTransport<UIMessageChunk, UIMessage>` |
| chatOptions | Optional | Options customising the chat-transport behaviour. | <Table id='ChatTransportOptions'/> |

</Table>

<Table id='ChatTransportOptions' hidden>

| Property | Description | Type |
| --- | --- | --- |
| prepareSendMessagesRequest | Hook to customise the HTTP POST body and headers before each send. Called with the conversation context (see <Table id='SendMessagesRequestContext'/>); returns the body and headers to use. Default sends all previous messages as `history`. | `(context: SendMessagesRequestContext) => { body?: Record<string, unknown>; headers?: Record<string, string> }` or Undefined |

</Table>

<Table id='SendMessagesRequestContext' hidden>

| Property | Description | Type |
| --- | --- | --- |
| chatId | Chat session ID (from `useChat`'s `id`). | String or Undefined |
| trigger | What triggered the request: `'submit-message'` for new messages, `'regenerate-message'` for regeneration. | `'submit-message'` or `'regenerate-message'` |
| messageId | The message ID for edit or regeneration. For regeneration, the assistant message to regenerate. For edits (`submit-message` with `messageId`), the user message being replaced. Undefined when submitting a new message. | String or Undefined |
| history | Previous messages in the conversation (context for the LLM). | `UIMessage[]` |
| messages | The new messages being sent. Empty for regeneration. | `UIMessage[]` |
| forkOf | The msg-id of the message being forked (regenerated or edited). | String or Undefined |
| parent | The msg-id of the predecessor in the conversation thread. | String or Undefined |

</Table>

### Returns <a id="create-chat-transport-returns"/>

`ChatTransport`

A `ChatTransport` adapter that satisfies Vercel's `useChat({ transport })` parameter shape.

## ChatTransport <a id="ChatTransport"/>

The adapter object Vercel's `useChat` expects as `transport`. Has two methods, both invoked by `useChat` internally; you typically pass the adapter and never call these directly.

### Send messages <a id="ChatTransport-sendMessages"/>

<MethodSignature>{`sendMessages(options: SendMessagesOptions): Promise<ReadableStream<UIMessageChunk>>`}</MethodSignature>

Send messages and return a streaming response of `UIMessageChunk` events. Called by `useChat` when the user submits a new message or requests a regeneration.

### Reconnect to a stream <a id="ChatTransport-reconnectToStream"/>

<MethodSignature>{`reconnectToStream(options: { chatId: string }): Promise<ReadableStream<UIMessageChunk> | null>`}</MethodSignature>

Reconnect to an existing streaming response. Returns `null` if no active stream exists for the specified chat session.

### Close the chat transport <a id="ChatTransport-close"/>

<MethodSignature>{`close(options?: CloseOptions): Promise<void>`}</MethodSignature>

Close the underlying [`ClientTransport`](/docs/ai-transport/api/javascript/client-transport), releasing all resources. Delegates to `ClientTransport.close()`; pass `{ cancel }` to publish a cancel before tearing down. When the `ChatTransportProvider` owns the lifecycle, it calls this automatically on unmount.

### Streaming flag <a id="ChatTransport-streaming"/>

<MethodSignature>streaming: boolean</MethodSignature>

Read-only flag indicating whether an own-turn stream is currently being consumed by `useChat`. Used internally by [`useMessageSync`](/docs/ai-transport/api/react/use-message-sync) to gate `setMessages` calls during active streams.

### Subscribe to streaming-state changes <a id="ChatTransport-onStreamingChange"/>

<MethodSignature>{`onStreamingChange(callback: (streaming: boolean) => void): () => void`}</MethodSignature>

Subscribe to transitions between streaming and idle. The callback fires with `true` when an own-turn stream starts and `false` when it ends. Returns an unsubscribe function. Used by `useMessageSync` to coordinate message-state writes with the streaming gate.

## Example

<Code>
```javascript
import * as Ably from 'ably';
import { createClientTransport, createChatTransport } from '@ably/ai-transport/vercel';
import { useChat } from '@ai-sdk/react';

const ably = new Ably.Realtime({ authUrl: '/auth' });
const channel = ably.channels.get('conversation-42');

const clientTransport = createClientTransport({ channel });
const chatTransport = createChatTransport(clientTransport, {
  prepareSendMessagesRequest({ chatId, trigger, history, messages, forkOf, parent }) {
    return {
      body: { chatId, trigger, history, messages, forkOf, parent, locale: 'en-GB' },
      headers: { 'x-tenant-id': tenantId },
    };
  },
});

// In a React component:
const { messages, sendMessage, stop } = useChat({
  id: 'conversation-42',
  transport: chatTransport,
});
```
</Code>
