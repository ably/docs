Ably Queues provide a queueing mechanism to integrate Ably with your external service. Queues are good for asynchronous machine-to-machine communication, where you want to decouple the producer and consumer of messages. Queues are ideal for asynchronous machine-to-machine communication. Queues follow a classic message queueing pattern where machines act as producers and consumers:

- @Producers@ := Publish messages via Ably channels.
- @Consumers@ := Consume messages from the queue.

<aside data-type='note'>
<p>"Limits":/docs/pricing/limits#integration apply to queues based on your account plan.</p>
</aside>

h2(#lifecycle). Queues lifecycle

Messages enter a queue through the following process:

* You publish a message to a channel using Ably's REST or Realtime protocols—you never publish directly to a queue.
* If a queue rule matches the name of the channel the message was published to, Ably applies the rule and identifies the corresponding queue.
* Ably then copies the message into the designated queue, making it available for consumption.

Messages published to a queue go through the following lifecycle:

* Ably stores each message by placing it at the back of the queue.
* Ably retrieves the oldest message from the front of the queue and delivers it to a consumer.
* Ably enforces a strict first-in, first-out (FIFO) policy to maintain message order.
* Ably ensures that each message is delivered to only one consumer, even if multiple consumers are connected.

Queues decouple producers and consumers in your system:

* Producers can publish messages without waiting for consumers, which keeps your system responsive even under heavy load.
* You can increase throughput by simply adding more consumers, as each one pulls messages independently and helps distribute work efficiently.
* Ably stores messages until a consumer acknowledges successful processing, so if a consumer crashes or disconnects, no data is lost.
* Ably preserves message order per channel when a single consumer processes the queue. However, if multiple consumers are used or messages are routed from multiple channels, ordering is only guaranteed per channel, not across channels.

h2(#Use-case). Use cases

The following are some common use cases for Ably Queues:

* Your servers process each published message by performing a unit of work, for example, generating an image and attaching it to an email.
* Ably delivers each message to only one consumer, no matter how many consumers listen.
* You scale your architecture horizontally by adding more consumer worker servers to handle realtime data processing.
* Your servers consume realtime data from channels without maintaining state; they don't track which channels or clients are active or share state between them.
* Ably builds a backlog of messages if your consumers fall behind or go offline, ensuring no data is lost.
* You provision the necessary queues ahead of time, for example, one queue for chat messages and another for analytics events.

h2(#prov). Provision a new Queue

All Ably accounts have access to Ably Queue functionality, however to get started you need to provision a physical queue and set up a queue rule to move data from channels into that queue.

You can provision one or more queues for your app, however by default each new app is provisioned without any queues. Once a queue is configured, you will need to set up a queue rule that enables messages to be republished from pub/sub channels into a physical queue.

<aside data-type='note'>
<p>Unlike Ably's global pub/sub channels, queues are region-specific. You choose a region when provisioning a queue, and messages are stored and replicated within that physical region. To reduce latency, select the region closest to your backend servers.</p>
</aside>

The following steps explain how to provision an Ably Queue:

* Go to your Ably "dashboard":https://ably.com/dashboard/any and select your app.
* Navigate to the Queues tab.
* Click *Provision a new Queue*.
* In the New Queue section, fill in the following fields:
** *Name* - Enter a unique name for your queue. This name, combined with your app ID, forms the full queue identifier.
** *Region* - Select the physical region where the queue will be hosted, for example, US East (Virginia). All queues are replicated across two datacenters in that region for high availability.
** *TTL (minutes)* - Set the time-to-live for messages. The default and maximum is 60 minutes. Messages that are not consumed within the TTL will be moved to the "dead letter Queue":/docs/integrations/queues#deadletter.
** *Max length* - Define the maximum number of messages the queue can retain. The default and maximum is 10,000. When the queue is full, the oldest message is moved to the dead letter queue to make space.
* Click *Create* to finish provisioning the queue.

<aside data-type='note'>
<p>A "dead letter Queue":/docs/integrations/queues#deadletter will be created automatically. It stores messages that exceed their TTL, are discarded when the queue is full, or fail to be consumed.</p>
</aside>

h3(#setup). Setup queue rules

After you provision a Queue, create one or more Queue rules to republish messages, presence events, or channel events from pub/sub channels into that queue. Queue rules can either be used to publish to internal queues hosted by Ably or "external streams":/docs/integrations/streaming, for example, Amazon Kinesis.

The following steps explain how to set up a queue rule:

* Go to your Ably "dashboard":https://ably.com/dashboard/any and select your app.
* Navigate to the Integrations tab.
* Click *New Integration Rule*.
* In the *Select your rule type* section, choose Queue.
* Under *Select service integration*, confirm Ably hosted AMQP and STOMP queues is selected.
* In the *Choose queue* dropdown, select the queue you want to send data to.
** If you haven't set up a queue yet, click *Setup a queue* to provision one.
* (Optional) Add custom Headers as key:value pairs to include metadata.
** Click *Another header* to add additional headers.
* In the "*Source*":/docs/integrations/webhooks#sources dropdown, select which type of Ably event should trigger the rule:
** Message
** Presence
** Channel Lifecycle
** Channel Occupancy
* (Optional) Add a "*Channel Filter*":/docs/integrations/webhooks#channel-filter using a regular expression to apply the rule to specific channels. Leave empty to apply to all.
* Choose an "*Encoding*":/docs/integrations/webhooks#encoding format:
** JSON
** MsgPack
* Click *Create* to finish setting up the rule.

npm install amqp-consume-cli -g

Once installed, go to your Ably "dashboard":https://ably.com/dashboard/any and retrieve an API key. Your root API key typically has access to subscribe to all queues.

The following explains the details you will need to use the CLI:

* The server endpoint host.
* The server endpoint port.
* The vhost, which is usually shared.

  --host [Server endpoint host] --port [Server endpoint port] \

h2(#amqp). Use AMQP to consume messages

Use the AMQP protocol if you're working with SDKs that prefer binary, broker native messaging. Ably Queues support AMQP 0.9.1 over TLS, allowing you to integrate securely with existing AMQP tooling.

The following steps explain how to consume messages from an Ably Queue using the AMQP protocol:

Choose any AMQP client that supports AMQP over TLS, and gather your connection details from the *Integrations* section of the Ably "Dashboard":https://ably.com/dashboard/any.

The following steps explain the details you will need to use the AMQP protocol:

- @Queue name@ := For example: UATwBQ:example-queue, formed from your app ID and the name you gave the queue.
- @Host@ := For example: @us-east-1-a-queue.ably.io@
- @Port@ := @5671@, the TLS-only port supported for secure AMQP consumption.
- @Vhost@ := Always shared
- @Username@ := The part before the colon in your API key. For example, @APPID.KEYID@ from @APPID.KEYID:SECRET@.
- @Password@ := The part after the colon in your API key. For example, @SECRET@ from @APPID.KEYID:SECRET@.
- @AMQP Client@ := Connect using any AMQP 0.9.1 compatible client that supports TLS.

The following example shows how to consume from a queue in Node.js:

```[nodejs]
const url = 'amqps://APPID.KEYID:SECRET@us-east-1-a-queue.ably.io/shared'
amqp.connect(url, (err, conn) => {
  if (err) { return handleError(err) }

  /* Opens a channel for communication. The word channel is overloaded
     and this has nothing to do with pub/sub channels */
  conn.createChannel((err, ch) => {
    if (err) { return handleError(err) }

    /* Wait for messages published to the Ably Queue */
    ch.consume('UATwBQ:example-queue', (item) => {
      let decodedEnvelope = JSON.parse(item.content)

      /* The envelope messages attribute will only contain one message. However,
         in future versions, we may allow optional bundling of messages into a
         single queue message and as such this attribute is an Array to support
         that in future */
      let messages = Ably.Realtime.Message.fromEncodedArray(decodedEnvelope.messages)
      messages.forEach((message) => {
        actionMessage(message)
      })

      /* ACK (success) so that message is removed from queue */
      ch.ack(item)
    })
  })
})
```

The Node.js example above shows how to consume messages from a queue using AMQP. Take note of the following:

* The queue rule wraps each message in an envelope (default behavior).
* The first step is to parse the envelope JSON.
* The @Message.fromEncodedArray@ method decodes the payload into standard "@Message@":/docs/api/realtime-sdk/types#message objects.
* While messages is always an @Array@, each envelope currently contains only one message.
* This structure supports future enhancements such as message bundling.

<aside data-type='note'>
<p>AMQP connections may disconnect during routine maintenance. Most AMQP clients automatically reconnect and retry with backoff and jitter. If your client does not handle reconnection, you should implement this logic in your application code.</p>
</aside>

h2(#stomp). Use STOMP to consume messages

Use the STOMP protocol if you prefer a simple, text-based approach for interacting with message brokers. STOMP clients can communicate with any broker that supports the protocol, making it a good fit for working with Ably Queues.

Choose any STOMP client that supports STOMP over TLS, and gather your connection details from the *Integrations* section of the Ably "Dashboard":https://ably.com/dashboard/any.

The following steps explain the details you will need to use the STOMP protocol:

- @Queue name@ := For @UATwBQ:example-queue@, formed from your app ID and the name you gave the queue.
- @Host@ := For example: @us-east-1-a-queue.ably.io@
- @Port@ := 61614, the TLS-only STOMP port (different from AMQP's port).
- @Vhost@ := Always shared
- @Username@ := The part before the colon in your API key. For example, @APPID.KEYID@ from @APPID.KEYID:SECRET@.
- @Password@ := The part after the colon in your API key. For example, @SECRET@ from A@PPID.KEYID:SECRET@.
- @STOMP Client@ := Connect using any STOMP client that supports TLS.

The following is a simple Node.js example using the @stomp/stompjs@ SDK:

```[nodejs]
const connectOptions = {
  'host': 'us-east-1-a-queue.ably.io',
  'port': 61614, /* STOMP TLS port */
  'ssl': true,
  'connectHeaders':{
    'host': 'shared',
    'login': 'APPID.KEYID',
    'passcode': 'SECRET'
  }
}

Stompit.connect(connectOptions, (error, client) => {
  if (err) { return handleError(err) }

  const subscribeHeaders = {
    /* To subscribe to an existing queue, /amq/queue prefix is required */
    'destination': '/amq/queue/UATwBQ:example-queue',
    'ack': 'client-individual' /* each message requires an ACK to confirm it has been processed */
  }
  /* Wait for messages published to the Ably Queue */
  client.subscribe(subscribeHeaders, (error, message) => {
    if (err) { return handleError(err) }

    /* STOMP is a text-based protocol so decode UTF-8 string */
    message.readString('utf-8', (error, body) => {
      if (err) { return handleError(err) }

      let decodedEnvelope = JSON.parse(item.content)

      /* The envelope messages attribute will only contain one message. However,
         in future versions, we may allow optional bundling of messages into a
         single queue message and as such this attribute is an Array to support
         that in future */
      let messages = Ably.Realtime.Message.fromEncodedArray(decodedEnvelope.messages)
      messages.forEach((message) => {
        actionMessage(message)
      })

      client.ack(message)
    })
  })
})
```

The node.js example above shows how to consume messages from a queue using STOMP. Take note of the following:

* The queue rule wraps each message in an envelope (default setting).
* The first step is to parse the envelope as JSON.
* Use the Message.fromEncodedArray method to decode the message into standard Message objects.
* The messages attribute is always an array, even though each envelope currently contains only one message.
* This format allows for future support of message bundling.

<aside data-type='note'>
<p>Using AMQP, you can consume multiple queues from a single connection and fetch more than one message at a time. </p>
</aside>

h2(#enveloped). Enveloped and non-enveloped messages

When you publish a message to a channel, Ably wraps it in an envelope before sending it to the queue. This envelope contains metadata about the message, such as the source channel and the app ID.

When you configure a queue rule, Ably gives you the option to envelope messages. This option is enabled by default. In most cases, using envelopes adds flexibility. Ably includes additional metadata in a portable format, such as the clientId of the publisher and the channel the message came from.

If performance is your priority, you can disable envelopes. In that case, Ably publishes only the message payload (data). This removes one layer of parsing but requires you to decode the raw payload yourself.

By default, Ably encodes messages sent to queues in JSON (a text format). However, you can choose to use "MsgPack":https://msgpack.org/, a binary format, when configuring your queue rules.

h2(#deadletter). Dead letter Queues

When you provision a Queue, Ably also provisions a dead letter Queue automatically. This special queue stores messages that fail processing or expire before being consumed.

We recommend consuming messages from the dead letter queue so you can monitor failures, expirations, and unprocessable messages in your system.

Ably moves messages into the dead letter queue in the following cases:
* The consumer rejects a message using @basic.reject@ or @basic.nack@ with @requeue=false@.
* The message exceeds its TTL and expires.
* The Queue reaches its maximum length and a new message is published. In this case, Ably removes the oldest message from the queue and places it in the dead letter queue to make room.

Ably deletes any message in the dead letter queue if it later meets one of these conditions—for example, if it expires due to TTL. These messages are not recoverable.

Ably names the dead letter queue using the reserved format @APPID:deadletter@, where APPID is your app's ID. Each app with one or more queues has exactly one dead letter queue, and you'll see it listed in the Queues dashboard. You can subscribe to it like any other queue.
