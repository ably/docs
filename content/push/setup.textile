Integrating push notifications into your application requires configuration. This configuration involves adjusting your app's platform for compatibility with Ably and activating devices for the relevant push notification services.

To integrate push notifications with Ably into your application, you need to configure your app's platform or OS to work compatibly with Ably and activate devices for the corresponding push notification services. Configuration involves setting up the "Apple Push Notifications (APNs)":https://developer.apple.com/notifications/ devices and "Firebase Cloud Messaging (FCM)":https://firebase.google.com/docs/cloud-messaging/ for your application to send and receive push notifications.

h2. Platform configuration

h3. Android configuration

1. Create an "FCM":https://firebase.google.com/docs/cloud-messaging/ project.
2. Set up "FCM":https://firebase.google.com/docs/cloud-messaging/android/client in your Android application. 
3. Create and add a "service account key":https://cloud.google.com/iam/docs/keys-create-delete#iam-service-account-keys-create-console to your Firebase project.
4. Log in to your "Ably dashboard":https://ably.com/accounts/any.
5. Click *Notifications*. Then click *Configure Push* in the push notification setup section.
6. Next, click the *FCM service account key* option
7. Then, enter your *service account key* and *Save*.

To receive messages on an Android device, implement "@FirebaseMessagingService@":https://firebase.google.com/docs/reference/android/com/google/firebase/messaging/FirebaseMessagingService. Override the @onMessageReceived()@ method to handle incoming push notifications.

Ensure your application handles push notification messages sent as @RemoteMessage@ instances via the Ably platform. Implement your own logic, as Ably clients do not automatically recognize these types of messages. For example, "display a notification":https://developer.android.com/develop/ui/views/notifications on the device to alert the user.

To handle push notifications sent as @RemoteMessage@ instances through the Ably platform, you must inform Ably clients by implementing the necessary logic. You can inform clients with actions such as displaying a notification on the device to alert users of new messages.

Ably uses the device’s registration token to push messages to devices. Keep Ably updated with the latest device registration token by overriding "@FirebaseMessagingService#onNewToken@":https://firebase.google.com/docs/reference/android/com/google/firebase/messaging/FirebaseMessagingService#public-void-onnewtoken-string-token and calling "@ActivationContext#onNewRegistrationToken@" and calling ActivationContext#onNewRegistrationToken:

```[android]
public class AblyPushNotificationService extends FirebaseMessagingService {
  public static final String PUSH_NOTIFICATION_ACTION = AblyPushNotificationService.class.getName() + ".PUSH_NOTIFICATION_MESSAGE";

  @Override
  public void onMessageReceived(@NonNull RemoteMessage message) {
      // TODO Handle the message delivered to the device.
  }

  @Override
  public void onNewToken(@NonNull String s) {
    super.onNewToken(s);
    //Store token in Ably
    ActivationContext.getActivationContext(this).onNewRegistrationToken(RegistrationToken.Type.FCM, s);
  }
}
```

h3. iOS configuration

1. Create an "APNs .p12":/:https://faqs.ably.com/how-do-i-obtain-the-apns-certificates-needed-for-ios-push-notifications.
2. Click the *P12* option in the *setting up Apple push notification service* section.
3. Next, *Upload file*.
4. Then, enter your *P12 password* and *Save*.

To receive messages on an Apple device call @UIApplication.shared.registerForRemoteNotifications@ in @UIApplicationDelegate#didFinishLaunchingWithOptions@. 

You need to override two methods in @UIApplicationDelegate@: <span lang="swift">"@application(_:didRegisterForRemoteNotificationsWithDeviceToken:)@":https://developer.apple.com/documentation/uikit/uiapplicationdelegate/1622958-application</span><span lang="objc">"@application:didRegisterForRemoteNotificationsWithDeviceToken:@":https://developer.apple.com/documentation/uikit/uiapplicationdelegate/1622958-application?language=objc</span> and <span lang="swift">"@application(_:didFailToRegisterForRemoteNotificationsWithError:)@":https://developer.apple.com/documentation/uikit/uiapplicationdelegate/1622962-application</span><span lang="objc">"@application:didFailToRegisterForRemoteNotificationsWithError:@":https://developer.apple.com/documentation/uikit/uiapplicationdelegate/1622962-application?language=objc</span>. Provide Ably with the @deviceToken@ and @error@ from the delegate methods by calling @ARTPush@, @didRegisterForRemoteNotificationsWithDeviceToken@ and @didFailToRegisterForRemoteNotificationsWithError@. You can provide either a rest or realtime client, which is used to authenticate with Ably.

```[objc]
    ARTClientOptions *options = [[ARTClientOptions alloc] init];

```

```[swift]
    var options = ARTClientOptions()

```

h2. Device activation

Activating a device to receive push notifications through Ably's infrastructure ensures that the device is correctly recognized and authenticated by the push notification service and Ably's platform.

You can a active your devices "directly":/#direct-activation or using "channels":/#channel-activation.

h3(#direct-activation). Direct activation

Activating push notifications directly from a device involves the application communicating with both the push notification service and the application's backend (Ably) to register for receiving push notifications. This direct approach allows applications to manage push notifications at the device level. The following illustration outlines the direct device activation process:

<a href="/images/diagrams/Ably-Push-Notifications-to-Device2.gif" target="_blank">
  <img src="/images/diagrams/Ably-Push-Notifications-to-Device2.gif" style="width: 100%" alt="Push Notifications direct device registration">
</a>

The following instructions explain how to activate your device directly:

1. The application requests permission from the user to receive push notifications, a crucial step for respecting user preferences and privacy.
2. The application interacts with the device's operating system to register with the appropriate push notification servics. This action generates a unique device token or identifier from the push service.
3. The application sends the unique device token to the application's backend server or a third-party push notification management service (Ably). This token is required to identify the device in future push notification requests.
4. The backend server or push notification service stores the device token, linking it with the user's account or device profile. This link allows the server to direct push notifications to specific devices based on user activities, preferences, or other criteria.
5. With the device token securely stored and associated, the device is prepared to receive push notifications. The server can send targeted messages to this device by including the stored token in notification requests to APNs or FCM.

h4. Activate using @push.activate@

To begin receiving push notifications from Ably in your main activity, you must initiate the process by calling "<span lang="android">@AblyRealtime.push.activate@</span><span lang="objc,swift">@ARTRealtime.push.activate@</span>":/api/realtime-sdk/push#activate.

The following instructions guide you through activating your device using the email address @push.activate@:

1. Ensure the Ably client is authenticated.
2. Generate a unique identifier for this device and store this in local storage.
3. Activate the device for push notifications with the underlying OS or platform and obtain a unique identifier for the device as a push recipient. For example, in FCM this is described as a "@registration token@":https://firebase.google.com/docs/cloud-messaging/android/client#sample-register, and in APNs this is described as a "@device token@":https://developer.apple.com/library/content/documentation/NetworkingInternet/Conceptual/RemoteNotificationsPG/CommunicatingwithAPNs.html.
4. Register the local device with Ably using the device's unique identifier, platform-specific details such as form factor and OS, and the push recipient details to receive push notifications. This in turn ensures Ably can reach this device and deliver push notifications.
5. Store the "@deviceIdentityToken@":/api/realtime-sdk/push#local-device from the response from Ably in local storage so that subsequent requests to Ably to update push recipient details are authenticated as being from the device in question.

The effects of calling "@activate@":/api/realtime-sdk/push#activate outlives the current process. Once called, the device will stay registered even after the application is closed, and up until "@deactivate@":/api/realtime-sdk/push#deactivate is called. "@activate@":/api/realtime-sdk/push#activate is idempotent: calling it again when device is already activated has the sole effect of calling its callback.

```[android]
AblyRealtime ably = getAblyRealtime();
ably.setAndroidContext(context)
ably.push.activate();
```

```[objc]
ARTRealtime *ably = [self getAblyRealtime];
[ably.push activate];
```

```[swift]
let ably = self.getAblyRealtime()
ably.push.activate()
```

For the client to register itself automatically with Ably, it needs to be authenticated and have "the required @push-subscribe@ capability":#required-capabilities. If you would prefer to delegate registration of the push device to your own servers and not allow devices to register themselves directly with Ably, then see the section "how to register devices from your server":#activation-from-server. You can also check "our recommendations":#server-vs-direct-registration for choosing either registration strategy.

h3(#channel-activation). Channel-based activation 

Activating devices for Push notifications through channels is a centralized approach for managing push notification subscriptions. This method handles device token management and push notification services interactions through your server. It's helpful in scenarios demanding enhanced control over the notification logic or where direct device registration is impractical.

<a href="/images/diagrams/Ably-Push-Notifications-Flow2.gif" target="_blank">
  <img src="/images/diagrams/Ably-Push-Notifications-Flow2.gif" style="width: 100%" alt="Push Notifications device registration via server">
</a>

The following instructions guide you through activating your devices using channels:

1. The application on the user's device must first gain permission to receive push notifications. After consent, the device registers with the applicable push notification service and acquires a unique device token or identifier.
2. The device forwards this identifier to your application's backend server. This transfer could coincide with the user's login, app startup, or another suitable occasion.
3. Once the server receives the device identifier, it stores and usually links it to the user's account or a specific app session, a vital step for directing notifications to the correct device.
4. With the device token on record, the server can register the device for push notifications on the user's behalf. This step involves Ably, which oversees the subscription and push notification distribution. The server indicates which devices, identified by the stored tokens, to activate for push notifications.
5. The server can dynamically adjust device subscriptions, adding or excluding devices from the notification roster as influenced by user activities, preferences, or operational logic, ensuring precise notification distribution management.

h4. Activate using @AblyRealtime.push.activate@

The default for @AblyRealtime.push.activate@ is to register the device with Ably directly from the device, but you can instead delegate that to your server. Don't forget to register the device using the "Push Admin API":/push/publish#push-admin in your server.

blang[objc,swift].
  @UIApplicationDelegate@ must implement these optional methods from @ARTPushRegistererDelegate@:

  ```[objc]
  - (void)ablyPushCustomRegister:(ARTErrorInfo *)error
                   deviceDetails:(ARTDeviceDetails *)deviceDetails
                        callback:(void (^)(ARTDeviceIdentityTokenDetails * _Nullable, ARTErrorInfo * _Nullable))callback {

          callback(nil, error);
          return;
      }

      [self registerThroughYourServer:deviceDetails callback:callback];
  }

  - (void)ablyPushCustomDeregister:(ARTErrorInfo *)error d
                          deviceId:(ARTDeviceId *)deviceId
                          callback:(void (^)(ARTErrorInfo * _Nullable))callback {
      if (error) {
          // Handle error.
          callback(nil, error);
          return;
      }

      [self unregisterThroughYourServer:deviceDetails callback:callback];
  }
  ```
  ```[swift]
  func ablyPushCustomRegister(_ error: ARTErrorInfo?, deviceDetails: ARTDeviceDetails, callback: @escaping (ARTDeviceIdentityTokenDetails?, ARTErrorInfo?) -> Void) {
      if let e = error {

          callback(nil, e)
          return
      }

      self.registerThroughYourServer(deviceDetails: deviceDetails, callback: callback)
  }

  func ablyPushCustomDeregister(_ error: ARTErrorInfo?, deviceId: String, callback: ((ARTErrorInfo?) -> Void)? = nil) {
  ```

blang[android].
  You need to communicate back and forth with the Ably library via the application's "@LocalBroadcastManager@":https://developer.android.com/reference/android/support/v4/content/LocalBroadcastManager. Ensure you pass @true@ as the @useCustomRegisterer@ parameter to "@activate@":/api/realtime-sdk/push#activate (and for "@deactivate@":/api/realtime-sdk/push#deactivate).

  ```[android]
  ably.push.activate(context, true);
  ably.push.deactivate(context, true);
  ```

  The Ably library will broadcast an @io.ably.broadcast.PUSH_REGISTER_DEVICE@ action when it needs you to register from your server, and @io.ably.broadcast.PUSH_DEREGISTER_DEVICE@ when it needs you to deregister. You must configure a listener to those actions in your application's @AndroidManifest.xml@, and from it answer back with a @PUSH_DEVICE_REGISTERED@ or @PUSH_DEVICE_DEREGISTERED@:

  ```[xml]
  <receiver android:name=".MyAblyBroadcastReceiver" >
    <intent-filter>
       <action android:name="io.ably.broadcast.PUSH_REGISTER_DEVICE" />
       <action android:name="io.ably.broadcast.PUSH_DEREGISTER_DEVICE" />
    </intent-filter>
  </receiver>
  ```

  ```[android]
  public class MyAblyBroadcastReceiver extends BroadcastReceiver {
      @Override
      public void onReceive(Context context, Intent intent) {
          AblyRealtime ably = getAblyRealtime();
          String action = intent.getAction();

          if (action.equals("io.ably.broadcast.PUSH_REGISTER_DEVICE")) {
              DeviceDetails device = ably.device(context);
              boolean isNew = intent.getBooleanExtra("isNew", false);

              Intent response = new Intent("io.ably.broadcast.PUSH_DEVICE_REGISTERED");

              try {
                  String deviceIdentityToken = registerThroughYourServer(device, isNew);
                  response.putExtra("deviceIdentityToken", deviceIdentityToken);
              } catch(AblyException e) {
                  IntentUtils.addErrorInfo(intent, e.errorInfo);
              }

              LocalBroadcastManager.getInstance(context.getApplicationContext()).sendBroadcast(intent);
          } else if (action.equals("io.ably.broadcast.PUSH_DEREGISTER_DEVICE")) {
              DeviceDetails device = ably.device(context);

              Intent response = new Intent("io.ably.broadcast.PUSH_DEVICE_DEREGISTERED");

              try {
                  deregisterThroughYourServer(device.id);
              } catch(AblyException e) {
                  IntentUtils.addErrorInfo(intent, e.errorInfo);
              }

              LocalBroadcastManager.getInstance(context.getApplicationContext()).sendBroadcast(intent);
          }
      }
  }
  ```

h2(#subscribing). Subscribe for push notifications

Subscribe a device directly using its "deviceID":#subscribing-device-id or through a channel using "clientID":#subscribing-client-id. 

h3(#subscribing-device-id). Subscribing by deviceID

A deviceID serves as a unique identifier for a device within Ably's services, automatically assigned when the device is activated.

You can execute the following if your client has the "push-subscribe":#push-capabilities capabilities:

```[objc]
[[realtime.channels get:@"pushenabled:foo"].push subscribeDevice:^(ARTErrorInfo *error) {
    // Check error.
}];
```

```[swift]
realtime.channels.get("pushenabled:foo").push.subscribeDevice { error
    // Check error.
}
```

```[android]
realtime.channels.get("pushenabled:foo").push.subscribeDevice(context);

// or

realtime.channels.get("pushenabled:foo").push.subscribeDeviceAsync(context, new CompletionListener() {
    @Override
    public void onSuccess() {}

    @Override
    public void onError(ErrorInfo errorInfo) {
        // Handle error.
    }
});
```

If your client doesn't have the @push-subscribe@ capabilities, you should communicate the deviceID to your server so that it can subscribe on the device's behalf. You can find your unique deviceID at "<span lang="objc,swift">@ARTRealtime.device.id@</span><span lang="android">@AblyRealtime.device().id@</span>":/api/realtime-sdk/push#device-details. The server must then use the "Push Admin API":/push/publish#push-admin to subscribe the device.

h3(#subscribing-client-id). Subscribing by clientID

When a device is registered, it can be associated with a "clientID":/auth/identified-clients. "<span lang="android">@AblyRealtime.push.activate@</span><span lang="objc,swift">@ARTRealtime.push.activate@</span>":/api/realtime-sdk/push#activate takes the clientID from the @AblyRealtime@ instance.

You can subscribe all devices associated with a clientID to a channel in one operation by creating a subscription by clientID. Any new device registrations linked to that clientID will automatically subscribe to the channel. If a device registration is no longer linked to that clientID, it will stop being subscribed to the channel, unless it is also directly subscribed by "deviceID":#subscribing-device-id. 

<aside data-type='note'>
<p>Before subscribing to a channel for Push notifications, you must explicitly configure its channel "namespace":/channels#namespaces to allow push notifications, as they are disabled by default and require activation. Push notifications are not enabled on channels by default.</p>
</aside>

Subscribe your @AblyRealtime@ instance's clientID to a channel:

```[objc]
[[realtime.channels get:@"pushenabled:foo"].push subscribeClient:^(ARTErrorInfo *error) {
    // Check error.
}];
```

```[swift]
realtime.channels.get("pushenabled:foo").push.subscribeClient { error
    // Check error.
}
```

```[android]
realtime.channels.get("pushenabled:foo").push.subscribeClient();

// or

realtime.channels.get("pushenabled:foo").push.subscribeClientAsync(new CompletionListener() {
    @Override
    public void onSuccess() {}

    @Override
    public void onError(ErrorInfo errorInfo) {
        // Handle error.
    }
});
```

Alternatively, if you want to subscribe a different clientID not currently associated with the currently authenticated realtime instance, you can use the "Push Admin API":/push/publish#push-admin.

h2. Handle push notification lifecycle

When the "@activate@":/api/realtime-sdk/push#activate and "@deactivate@":/api/realtime-sdk/push#deactivate methods are called, the device registers with FCM/APNs, initiating activation with Ably. Moreover, suppose a push token, either an APNs device token or an FCM registration token, is updated on the device. In that case, the Ably SDK attempts to update Ably's servers with this new token. It's important to note that the activation, deactivation, and update processes may fail, so managing the outcome of these methods within the push notification lifecycle methods is crucial.

To monitor the status of push notification operations in your application, you should listen to three specific broadcast intents that correspond to the different events in the process:

1. Push activation completion (Success or Failure). Listen for the @io.ably.broadcast.PUSH_ACTIVATE@ action in a broadcast intent. Once the device has been successfully activated, you can subscribe to push notifications on channels to receive push notifications from Ably.
2. Push deactivation completion (Success or Failure). Listen for the @io.ably.broadcast.PUSH_DEACTIVATE@ action in a broadcast intent.
3. Push update completion (Failure only). Listen for the @io.ably.broadcast.PUSH_UPDATE_FAILED@ action in a broadcast intent.

The following example demonstrates listening to the push activation event:

```[android]
LocalBroadcastManager.getInstance(context.getApplicationContext()).registerReceiver(new BroadcastReceiver() {
    @Override
    public void onReceive(Context context, Intent intent) {
        ErrorInfo error = IntentUtils.getErrorInfo(intent);
        if (error != null) {
            // Handle error
            return;
        }
        // Subscribe to channels / listen for push etc.
    }
}, new IntentFilter("io.ably.broadcast.PUSH_ACTIVATE"));

ably.push.activate(context);
```

```[objc]
// Add the activate method from 'ARTPushRegistererDelegate' to your 'UIApplicationDelegate' class:

// Call activate, which will call the delegate method when done:

```
