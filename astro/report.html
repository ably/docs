Why this matters now

Gatsby entered maintenance mode under its current steward. It's shipped no meaningful feature releases since 2023, and is increasingly out of step with how the rest of the web is built. Our docs site is the primary organic-search surface for Ably — every extra second of load time is revenue attribution lost, and every minute spent waiting on Gatsby's build is author time not spent writing. The PoC shows the migration is straightforward, measurably dramatic in its gains, and does not require rebuilding anything customers see today.

What this means in practice

Author velocity

Content iteration goes from coffee-break to real-time

Writers see their changes in under a second on save instead of waiting 15–30 s each cycle. Over a writing session that's an extra 1–2 hours of throughput per author per day — measured against current dev-server wait times.

Reader experience

Every content page becomes instant

Prose-heavy pages ship zero JavaScript. First paint is the HTML the browser already has. Google ranks pages by Core Web Vitals and competitors on modern stacks are pulling ahead — this closes the gap decisively for docs SEO.

CI / infrastructure

Every PR builds 4–8× cheaper

Heroku and CI minutes scale with build time. Every PR to the docs repo currently burns 5–8 minutes of Gatsby build time multiplied by parallel preview deploys. Cutting that to ~60–120 seconds is a proportional cost and time saving for every change.

Hiring & retention

Modern stack, meaningful CV

Frontend engineers increasingly expect to work on Astro, Vite, or Next. Gatsby is no longer on anyone's "learn this next" list. A current stack is a real factor in who we can hire and retain on the docs-adjacent teams.

Strategic risk

Gatsby's trajectory is down

Netlify pivoted its strategy after acquiring Gatsby. Releases are rare, the Slack community is quiet, and plugin authors are drifting to other ecosystems. Staying on Gatsby is increasingly a one-way street — each year the migration cost grows, not shrinks.

User-perceived quality

The docs site will feel faster

View Transitions give SPA-like nav continuity — chrome stays alive page-to-page, no re-hydration flicker. Combined with zero-JS content and Lighthouse 95+ scores, the site becomes tangibly sharper. Customers notice.

Why Astro and not Next.js

We seriously considered Next.js as the migration target. It's the default answer in many rooms. The conclusion: Next is the right choice for app-like workloads (authenticated dashboards, server-rendered personalised content, streaming UI with RSC). Astro is the right choice for content-first sites (docs, blogs, marketing) where most pages are static prose. Ably Docs is a content-first site.

When we would pick Next.js instead

If Ably Docs grew into an app-like surface — authenticated per-customer content, personalised code samples tied to a logged-in account, an integrated support inbox, live API explorers rendering per-request — Next.js would become the right answer. The current docs site is not that. If that future arrives, Astro's islands model + Node adapter can absorb a surprising amount of app-like surface before a Next rewrite pays for itself.

What the PoC proved

Every concrete risk we identified up-front has either been eliminated or de-risked to a known workaround. The migration is an easy one by design: preserve the heavier React aspects (Header, Footer, LeftSidebar, RightSidebar, PageHeader) unchanged by lifting them into Astro as React islands; streamline the lighter content aspects (MDX, remark plugins, content pipeline) using Astro-native primitives.

  • MDX content reads in place from the existing src/pages/docs/ tree via Astro's content-layer glob loader. No copy, no symlink, no shape change.
  • Real Header/Footer/Sidebars render under Astro unchanged — a thin gatsby + @reach/router shim plus a Vite alias was enough. No rewrite.
  • Redux is gone. A 610-byte UserProvider island + two nanostores replace the entire createRemoteDataStore + connectState setup.
  • <If> retired — all 1,200+ call sites compile to CSS-gated <div data-*-gate> wrappers via a remark plugin. Zero hydration cost per gate, instant CSS-only toggle on language/auth change.
  • SPA-style navigation via Astro's <ClientRouter /> + transition:persist on chrome islands. Header/Footer/LeftSidebar survive page-to-page nav without re-hydrating.

Honest costs

What we gain

  • Faster everything. 4–8× build, 40× dev start. Authoring feedback loop turns from "coffee-break" into "real-time".
  • Zero-JS content pages. The 200+ prose-heavy pages in docs ship no JavaScript at all.
  • Simpler mental model. No GraphQL data layer. Content is TypeScript imports. Plugins are small remark/rehype functions authored in-tree.
  • Active framework. Astro ships Server Islands, Content Layer, View Transitions, Actions in the last year. Gatsby hasn't had a meaningful feature release since 2023.
  • SPA-like nav. Chrome persists across page transitions — a UX improvement over the current stack.

What it costs us

  • Astro 6 is blocked by @ably/ui shipping ESM with extensionless relative imports. Fix is one-line upstream. Until then we stay on Astro 5 (still actively patched).
  • Tailwind 3 lock. The @ably/ui preset is Tailwind 3–shaped; Tailwind 4 requires a separate preset rewrite. Independent of this migration.
  • Chrome bundle is real. The lift-and-shift approach pulls Radix, @ably/ui, lodash, es-toolkit transitively — roughly 220 kB gz production. Code-split per route mitigates; full retirement of @ably/ui is a separate workstream.
  • Real CodeSnippet needs upstream work. Astro's MDX wraps fenced blocks in an opaque <StaticHtml>; @ably/ui's CodeSnippet walks React children to detect language and gets nothing. Workaround matches the visual; full fidelity needs a prop-based API upstream.
  • Hydration-mismatch pitfalls. SSR-rendered React islands that read window.location need careful fallbacks. Mapped and workable.

Known risks / blockers

Watch items

These are real but bounded. None are reasons to stop; all are things the project plan needs to budget time for.

  • @ably/ui ESM extensionless imports — blocks Astro 6 today. One-line fix in @ably/ui's build config. Until then, Astro 5 is the target (actively supported).
  • Real CodeSnippet can't introspect MDX-wrapped fenced blocks — workaround in PoC matches the visual. Full fidelity (line numbers, language tabs, copy button) needs a prop-based CodeSnippet variant in @ably/ui.
  • Tailwind 4 adoption is out of scope for this migration. @ably/ui would need to ship a Tailwind 4–shaped preset first. Independent project.
  • Content drift during the port — if authoring continues on Gatsby while migration is underway, merge effort grows proportionally. Mitigate with a freeze window or frequent rebases.
  • Inkeep search + Ask AI integration are behind feature flags in the PoC — minor wiring, worth allocating time for once the port lands.

Strategic framing

The case for timing

Docs platform migrations get harder every quarter they're deferred. Content volume grows, Gatsby plugin compatibility narrows, engineer familiarity with the stack fades. The PoC demonstrates the migration is technically straightforward today; the same cannot be assumed indefinitely. Choosing to modernise now locks in the performance, SEO, and velocity gains above, and removes a tech-debt line item that will otherwise keep accruing interest.
