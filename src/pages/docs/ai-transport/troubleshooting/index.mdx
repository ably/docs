Troubleshooting is organised by subsystem. Each page covers symptoms, what the root cause usually is, and how to confirm and fix.

- [Connection issues](/docs/ai-transport/troubleshooting/connection): channel attach failures, reconnect loops, message delivery gaps.
- [Authentication errors](/docs/ai-transport/troubleshooting/auth): token rejected, capability missing, `clientId` mismatch.
- [Codec problems](/docs/ai-transport/troubleshooting/codec): events dropped, stream parts missing, accumulator mismatches.
- [Turn lifecycle](/docs/ai-transport/troubleshooting/turns): turns that never start, never end, or end with the wrong reason.

Each feature page also has its own FAQ that covers questions specific to that feature. Start there if you have a feature-specific symptom.

## How to use these pages <a id="how-to-use"/>

1. Identify the subsystem most likely involved. If the client cannot connect, it is connection or auth. If the conversation tree looks wrong, it is the codec or the wire protocol. If a stream starts but does not finish, it is turn lifecycle.
2. Find the matching symptom on the subsystem page.
3. Follow the confirmation step (often a grep or a console command) before applying the fix. The fastest fix for the wrong cause loses you more time than the confirmation took.

## When to escalate <a id="escalate"/>

If you have walked through the relevant subsystem page and the symptom does not match, or the fix did not work, capture:

- The channel name.
- The date and time.
- The client ID of the affected server agent or client device.
- The first error log entry on either side.

[Open a support ticket](https://ably.com/contact) with these. The Ably side of the system is observable to support; the application side needs the IDs to correlate.
