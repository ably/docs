There are two ways to publish and receive push notifications using Ably:

- Direct publishing:= delivers notifications directly devices or groups. You can send personalized messages or notifications based on unique identifiers like "deviceID":/push/setup#subscribing-device-id or "clientID":/push/setup#subscribing-client-id, making migration seamless for apps transitioning to Ably with existing recipients.
- Channel-based publishing:= delivers push notifications to channel subscribers. Channel-based publishing mirrors Ablys "Pub/Sub channels":/channels, an architectural pattern that enables publishers to send messages to a channel and any channel subscribers to receive them.

Both approaches utilize a "push notification payload structure":#payload-structure, incorporating the notification’s content, and the data the application requires. You can customize the payload for iOS and Android platforms to enhance the user experience. 

h2(#payload-structure). Push notification payload structure

The push notification payload has a generic structure:

```[json]
{
  "notification": {
    "title": <string, title to display at the notification>,
    "body": <string, text below title on the expanded notification>,
    "icon": <string, platform-specific>,
    "sound": <string, platform-specific>,
    "collapseKey": <string, platform-specific, used to group notifications together>
  },
  "data": {
    <key string>: <value string>,
    ...
  }
}
```

Ably automatically adjusts message fields to ensure compatibility with the target push notification transport:

<!--[!!Ed. note: Removed experimental component column] _=. Web (*experimental*) | @notification.title@ | @notification.body@ |@notification.icon@ |@notification.sound@ |@notification.collapseKey@ |@data@ |-->

|_=. Ably field |_=. FCM |_=. APNs |
| @notification.title@ | @notification.title@ | @aps.alert.title@ |
| @notification.body@ | @notification.body@ | @aps.alert.body@ |
| @notification.icon@ | @notification.icon@ | Discarded. |
| @notification.sound@ | @notification.sound@ | @aps.alert.sound@ |
| @notification.collapseKey@ | @collapse_key@ | @aps.thread-id@ |
| @data@ | @data@ | Merged into root object. |

The following example is a push payload, in a message, published to Ably:

```[json]
{
  "notification": {
    "collapseKey": "chat"
  }
}
```

Raw format to FCM:

```[json]
{
  "collapse_key": "chat"
}
```

Raw format to APNs:

```[json]
{
  "aps.thread-id": "chat"
}
```
<!--[!!Ed. note: reformat code if this is to be returned to production docs.] and would be delivered in raw format to a web push target (*experimental*) as: ```[json]{  "notification": {    "collapseKey": "chat"  }}```-->

You can set transport-specific attributes which will get merged into the root object resulting from generic mapping explained above only when pushing to the selected transport:

You must override the generic value to the field and add specific fields that are not supported in the generic structure.

Alongside @notification@ and @data@, add an object thats field is one of:

* @fcm@, for "FCM":https://firebase.google.com/docs/cloud-messaging/concept-options.
* @apns@, for "APNs":https://developer.apple.com/library/archive/documentation/NetworkingInternet/Conceptual/RemoteNotificationsPG/APNSOverview.html#//apple_ref/doc/uid/TP40008194-CH8-SW1.

<!--[!!Ed. note: removed as part of EDU-1155. Return if required in the future.] * @web@, for Web Notifications (*experimental*).-->

The following example uses the payload structure to send a push notification:

```[json]
{
  "notification": {
    "title": "Hello from Ably!",
    "body": "Example push notification from Ably."
  },
  "data": {
    "foo": "bar",
    "baz": "qux"
  },
  "apns": {
    "aps": {
      "alert": {
        "title": "Hello to iOS from Ably!"
      }
    }
  },
  "fcm": {
    "notification": {
      "color": "#d3d3d3"
    }
  }
}
```
It includes

* A general notification structure with a title and body for the notification.
* A data section containing key-value pairs for additional information.
* An `apns` field for Apple Push Notification Service (APNS) specific settings, including an alert title for iOS devices.
* An `fcm` field for Firebase Cloud Messaging (FCM) with notification customization options, such as color.

This structure allows for targeted notifications across different platforms (iOS, Android) with custom data and appearance settings.


h3(#apns-headers). APNs Headers

When sending background notifications on iOS, include specific headers in the notification. You can do this by adding an @apns-headers@ object that contains all desired APNs headers as part of the @apns@ object:

```[json]
{
  "data": {
    "foo": "bar",
    "baz": "qux"
  },
  "apns": {
    "aps": {
      "content-available": 1,
    },
    "apns-headers": {
      "apns-push-type": "background",
      "apns-priority": "5"
    }
  }
}
```

h2(#direct-publishing). Direct publishing

Direct Push notifcation management involves individual devices handling push notification settings and subscriptions through "Ably's client SDKs":https://ably.com/docs/sdks. Devices must register with the push notification service specific to their operating system and then activate themselves for Push notifications via the Ably SDK. Devices can subscribe to their channels to receive push notifications publish over those channels. Note, direct device management limitations are device specific. You can deliver Push notifications directly to:  

* Devices identified by their unique "deviceID":#device-id.
* Devices identified by their assigned "clientId":#client-id.
* Devices identified by their recipient attributes such as their unique @registrationToken@ in the case of FCM, or @deviceToken@ in the case of APNs.
<!--[!!Ed. note: removed as this is experimental. Kept in case it is required in the future.] or @targetUrl@ and @encryptionKey@ in the case of a Web device (*experimental*). -->
 This is useful when migrating to Ably with existing push notification target devices.

<aside data-type='note'>
<p>If @clientOptions@ is not configured with an @ARTPushRegistererDelegate@ through its @pushRegistererDelegate@ property, the @UIApplicationDelegate2@ is automatically utilized if it implements @ARTPushRegistererDelegate@. This fallback to @UIApplicationDelegate@ is the now deprecated method. </p>
</aside>

See "Push Admin":/api/realtime-sdk/push-admin#publish for the client library API details, and "Push publish REST API":/api/rest-api#push-publish for information on the underlying direct publishing endpoint used by the client libraries.

h2(#channel-publishing). Channel-based publishing

Ably's model intentionally mirrors the delivery of standard messages to realtime subscribers using "pub/sub channels":/channels for pushing notifications to devices over channels. For instance, publishing a standard message on an Ably channel immediately sends it to all channel subscribers. However, the process remains the same when publishing push notifications on channels, except that subscribers (devices receiving push notifications) must register in advance through our API. Additionally, the message must include an extra push notification payload specifying the push notification's optional visual format and optional data payload.

To publish push notifications on a channel, the following requirements must be met: 

* The *extra push notification* payload is included in the published message.
* A "channel rule":/channels#rules is configured explicitly enabling push notifications on that channel.
* The device subscribed to the channel.
* The push notification payload is compatible with the subscribed push notification device.

Push notifications are sent as special payloads alongside "a normal Ably message":/channels/messages in the @extras@ field. The @extras@ field is an object and must contain a @push@ attribute object with the push payload details.

```[objc]
ARTMessage *message = [[ARTMessage alloc] initWithName:@"example" data:@"rest data"];
message.extras = @{

        @"notification": @{
            @"title": @"Hello from Ably!",
            @"body": @"Example push notification from Ably."
        },
        @"data": @{
            @"foo": @"bar",
            @"baz": @"qux"
        }
    }
};
[[rest.channels get:@"pushenabled:foo"] publish:@[message]];
```

The "Push Admin API":/api/realtime-sdk/push-admin offers three key features:

* "@publish@":/api/realtime-sdk/push-admin#publish for direct publishing to devices and groups of identified devices.

h2(#push-admin). Push Admin API

Ably's "Push Admin API":/api/realtime-sdk/push-admin is a set of functionalities designed for backend servers to manage push notification tasks comprehensively. This API includes registering devices for push notifications, managing subscriptions to specific channels, and directly sending push notifications to devices or users identified by a client identifier. The Push Admin API is accessible through Ably's realtime or REST client interfaces, allowing seamless integration into server-side applications. It enables precise control over the push notification process, from device registration and subscription management to the actual delivery of notifications, ensuring targeted and efficient communication with users across platforms.

Using the "Push Admin API":/api/realtime-sdk/push-admin requires specific permissions as part of a client's credentials:

* Access using the @push-admin@ permission. A client whose credentials contain the @push-admin@ permission has full access to the API and can manage registrations and subscriptions for all devices. 

* Access using the @push-subscribe@ permission. A client with @push-subscribe@ is a push target device, and it can manage its registration and any subscription for itself; it is not able to manage push registrations or channel subscriptions for any other device. The credentials presented, as well as containing the @push-subscribe@ permission, must also authenticate the device itself.

Every push target device has a deviceID and credentials, which are used as the basis for device authentication. When using the push admin API, a device can authenticate itself in two ways:

* Using an "Ably token":/auth/token that contains its @deviceId@.

* Using a normal "Ably key":/auth/basic or "Ably Token":/auth/token but additionally including a @deviceIdentityToken@ - a credential created at registration time that is used to assert the device's identity - in a header in the request.

The client library performs management of device credentials, so unless the push admin API is being accessed directly using HTTP, the client application does not need to worry about managing device credentials.
