Before a client or server can issue requests to Ably, such as subscribe to channels, or publish messages, it must authenticate with Ably.

## Recommended authentication <a id="recommended"/>

| Environment | Recommended Method | Details |
|-------------|-------------------|---------|
| **Client-side** (browsers, mobile apps) | [JWT token authentication](/docs/platform/auth/token#choosing-jwt) | Your server creates JWTs; clients use `authCallback` to fetch them |
| **Server-side** (Node.js, Python, etc.) | [Basic authentication](/docs/platform/auth/basic) | Use your API key directly in trusted environments |

<Aside data-type='important'>
**For client-side applications, [JWT is the recommended token mechanism](/docs/platform/auth/token#choosing-jwt).** JWTs require no Ably SDK on your server, provide exclusive features like channel-scoped claims and per-connection rate limits, and work with any JWT library.
</Aside>

### Quick start: JWT authentication

Your server creates JWTs that clients fetch via `authCallback`:

<Code>
```javascript
// Client-side: fetch JWT from your server
const realtime = new Ably.Realtime({
  authCallback: async (tokenParams, callback) => {
    try {
      const response = await fetch('/api/ably-token');
      const jwt = await response.text();
      callback(null, jwt);
    } catch (error) {
      callback(error, null);
    }
  },
});
```
</Code>

See [token authentication](/docs/platform/auth/token) for complete server-side JWT creation examples in all languages.

## Authentication terminology <a id="terminology"/>

- **Authentication**: The process of verifying credentials (API key or token) to allow interaction with Ably.
- **Authorization**: Determining what operations an authenticated client can perform, based on [capabilities](/docs/platform/auth/capabilities).
- **Identified client**: A client with a verified identity (`clientId`). See [identified clients](/docs/platform/auth/identified-clients).

<Aside data-type='note'>
Channels can be configured to only allow [identified clients](/docs/platform/auth/identified-clients). Learn more about [channel rules](/docs/channels#rules).
</Aside>

## Ably API keys <a id="api-keys"/>

Every Ably app can have one or more API keys associated with it in order to authenticate directly with Ably, or to issue tokens with. API keys can be created with different [capabilities](/docs/platform/auth/capabilities) and any tokens issued using that API key can only permit a subset of those capabilities.

### API key format <a id="format"/>

An Ably API key string has the following format: `I2E_JQ.OqUdfg:EVKVTCBlzLBPYJiCZTsIW_pqylJ9WVRB5K9P19Ap1y0`.

The API key is made up of three parts:

1. `I2E_JQ` is the public app ID (the part before the first period)
2. `OqUdfg` is the public app key ID (the part after the period and before the colon). `I2E_JQ.OqUdfg` is the public API key ID (both the public app ID and app key ID together)
3. `EVKVTCBlzLBPYJiCZTsIW_pqylJ9WVRB5K9P19Ap1y0` is the API key secret and should never be shared with untrusted parties (the part after the colon)

The API key secret is private and should never be made public. This API key string is used in all Ably SDKs and for authentication with the REST API.

### Create an API key <a id="create"/>

API keys are created in the [Ably dashboard](https://ably.com/dashboard). You can also create an API key programmatically using the [Control API](/docs/platform/account/control-api).

To create an API key in the Ably dashboard:

1. In your [Ably dashboard](https://ably.com/dashboard) click the API Keys tab.
2. Click the **Create a new API key** button.
3. Enter a name for your API key - this will help you identify the specific key when you have many keys created.
4. Select the [capabilities](/docs/platform/auth/capabilities) you want to apply to the key. Clients connecting with this key will be restricted to these capabilities.
5. Optionally you can select whether to make tokens generated from this key to be revocable or not.
6. Optionally select whether you want to restrict the scope of the key to channels, queues, or specific channels and queues using resource names and wildcards.

<Aside data-type="important">
Many of the code samples used throughout the documentation use basic authentication to demonstrate features for convenience. It should be stressed that in your own applications, you should never use [basic authentication](/docs/platform/auth/basic) on the client-side, as it exposes your Ably API key.

If you are logged into your Ably account, the code samples will be populated with your actual key for convenience. If you are not logged into Ably, a temporary key is generated for you.
</Aside>

## Authentication mechanisms <a id="mechanisms"/>

Ably supports two authentication mechanisms:

1. **[Token authentication](/docs/platform/auth/token)** (recommended for clients): Short-lived tokens that expire and can be revoked. **[JWT is the recommended token type](/docs/platform/auth/token#choosing-jwt).**
2. **[Basic authentication](/docs/platform/auth/basic)** (server-side only): Uses your API key directly.

### Client-side: Use JWT token authentication <a id="client"/>

**Never use API keys in client-side code.** API keys don't expireâ€”once compromised, they grant indefinite access to your Ably account. Instead, use [JWT token authentication](/docs/platform/auth/token#choosing-jwt):

- Tokens expire automatically (typically 1 hour)
- Tokens can be [revoked](/docs/platform/auth/revocation) if compromised
- [Capabilities](/docs/platform/auth/capabilities) restrict what each token can do
- JWT-exclusive features: channel-scoped claims, per-connection rate limits

### Server-side: Use basic authentication <a id="server"/>

Use [basic authentication](/docs/platform/auth/basic) with your API key on trusted servers. Token authentication adds unnecessary overhead server-side since the server would need to periodically refresh tokens to authenticate itself.

## Selecting an authentication mechanism <a id="selecting-auth"/>

When deciding on which authentication method you will be using, it is recommended you bear in mind the [principle of least privilege](https://en.wikipedia.org/wiki/Principle_of_least_privilege).

A client should ideally only possess the credentials and rights that it needs to accomplish what it wants. This way, if the credentials are compromised, the rights that can be abused by an attacker are minimized.

The following table provides guidelines on what to consider when choosing your authentication method. Many applications will most naturally use a mixed strategy: one or more trusted application servers will use basic authentication to access the service and issue tokens over HTTPS, whereas remote browsers and devices will use individually issued tokens:

| Scenario | [Basic](/docs/platform/auth/basic) | [Token](/docs/platform/auth/token) | Description |
|----------|---------------------------|---------------------------|-------------|
| Your scripts may be exposed | No | Yes | If the script, program or system holding the key is exposed, for example on a user's device, you should not embed an API key and instead use token authentication. If the script is on a secure environment such as your own server, an API key with basic authentication is fine. |
| Your connection may be insecure | No | Yes | If there is a risk of exposure of the client's credentials, either directly or over an insecure, or insecurely proxied, connection, token authentication should be used. If you are sure the connection is secure and unmediated, basic authentication is acceptable. |
| You have no server to control access | Yes | No  | If you do not have your own server to perform authentication and provide tokens to users, you'll need to use basic authentication. |
| You require fine-grained access control | No | Yes | If you need to provide [privileges](/docs/platform/auth/capabilities) on a user-by-user basis, you'd be better using token authentication. If you only need a few access control groups, basic authentication is reasonable. |
| Users need restricted periods of access | No | Yes | If you need users to only have access for a certain period of time, or the ability to revoke access, token authentication is needed. If users are able to always have access, basic authentication is acceptable. |
| Users need to identify themselves | Partial  | Yes | If the user can be trusted to [identify](/docs/platform/auth/identified-clients) itself, basic authentication is fine. If the user cannot be trusted however, token authentication is better as it allows for a trusted token distributor to identify the user instead. |
