Publishing is the process that sends Push notifications from the source the client application or server to the end-user's device. Ably supports two methods of publishing: Direct publishing and Channel-based publishing.

"Direct publishing":link sends notifications directly to individual devices without Channels. Each notification targets a specific device, akin to sending a personal message. This method is useful for sending personalized or specific notifications to individual users.

In "Channel-based publishing":link, Ably automatically sends Push notifications to multiple devices through Channels. Ably ensures that messages sent to a channel are delivered to all devices subscribed to it. This method is beneficial for sending the same notification to a group of devices, such as users subscribed to a particular topic or group.

Both of these methods use the "Push notification payload structure":#payload-structure, which includes the notification's content and any additional data required by the application. You can customize this payload structure for iOS and Android platforms to enhance the user experience.

h2(#channel-pub). Channel-based publishing

Channel-based publishing extends Ably's realtime messaging infrastructure to deliver notifications to devices via Channels, providing the necessary configurations and API support. Additionally, Push notifications require an extra Push notification payload to specify the notification's optional visual format and data payload.

The process of channel-based publishing is as follows:
1. Set up one or more Push subscriptions for the channel. These subscriptions can either be by "deviceID":#device-id for individual devices or "clientId":#client-id to encompass all devices associated with that clientID.
2. Publish a message on the channel and a Push notification payload.

Note that a device will receive a Push notification sent on a channel only if:
1. The published message includes the extra Push notification payload.
2. A channel rule explicitly enables Push notifications for that channel.
3. The device is subscribed to the channel.
4. The Push notification payload is compatible with the subscribed Push notification device.

The following is an example Push notifications sent as payloads alongside an Ably "message"::/channels/messages in the @extras@ field. The @extras@ field is an object and must contain a Push attribute object with the Push payload details:

```[jsall]
      title: 'Hello from Ably!',
      body: 'Example push notification from Ably.'
    },
    data: {
      foo: 'bar',
      baz: 'qux'
    }
  }
};

const channel = rest.channels.get('pushenabled:foo');
await channel.publish({ name: 'example', data: 'data', extras: extras });
```

h2(#direct-publishing). Direct publishing

Direct publishing sends notifications directly to individual devices using the "Ably SDK"::https://ably.com/docs/sdks. This process requires devices to register with Push notification services based on their operating system and then activate Push notifications through the Ably SDK. It's important to note that the limitations of direct publishing can vary depending on the specific device. Push notifications can be delivered directly to devices identified by their unique "deviceID":#device-id, their assigned "clientId":#client-id, or their "recipient attributes":#recipient. Direct publishing is also beneficial when migrating to Ably while targeting existing Push notification devices.

See "Push Notifications Admin":/api/realtime-sdk/push-admin#publish for the client library API details, and "Push publish REST API":/api/rest-api#push-publish for information on the underlying direct publishing endpoint used by the client libraries.

h3(#device-id). Direct publish using deviceID

Publish Push notifications using the deviceID:

* Structure your Push notification payload according to your requirements, including fields like notification title, body, and any additional data you want to send.
* Set the recipient of the Push notification. Specify the deviceID as the recipient.
* Use the Ably API to publish the Push notification with the specified recipient and payload.

h2(#payload-structure). Push notification payload structure

Ensure successful delivery, customization, and compatibility across various devices and operating systems using platform-specific requirements and best practices:


```[json]
{
  "notification": {
    "title": <string, title to display at the notification>,
    "body": <string, text below title on the expanded notification>,
    "icon": <string, platform-specific>,
    "sound": <string, platform-specific>,
    "collapseKey": <string, platform-specific, used to group notifications together>
  },
  "data": {
    <key string>: <value string>,
    ...
  }
}
```

Ably automatically adjusts message fields to ensure compatibility with the target Push notification transport:

<!--[!!Ed. note: Removed experimental component column] _=. Web (*experimental*) | @notification.title@ | @notification.body@ |@notification.icon@ |@notification.sound@ |@notification.collapseKey@ |@data@ |-->

|_=. Ably field |_=. FCM |_=. APNs |
| @notification.title@ | @notification.title@ | @aps.alert.title@ |
| @notification.body@ | @notification.body@ | @aps.alert.body@ |
| @notification.icon@ | @notification.icon@ | Discarded. |
| @notification.sound@ | @notification.sound@ | @aps.alert.sound@ |
| @notification.collapseKey@ | @collapse_key@ | @aps.thread-id@ |
| @data@ | @data@ | Merged into root object. |

You can use different payload formats for FCM and APNs when publishing messages to Ably. The following are examples push payloads in both formats:

```[json]
{
  "notification": {
    "collapseKey": "chat"
  }
}
```

Raw format to FCM:

```[json]
{
  "collapse_key": "chat"
}
```

Raw format to APNs:

```[json]
{
  "aps.thread-id": "chat"
}
```


Add FCM or APNs objects alongside the general notification and data fields to customize payloads for specific services:

```[json]
{
  "notification": {
    "title": "Hello from Ably!",
    "body": "Example Push notification from Ably."
  },
  "data": {
    "foo": "bar",
    "baz": "qux"
  },
  "apns": {
    "aps": {
      "alert": {
        "title": "Hello to iOS from Ably!"
      }
    }
  },
  "fcm": {
    "notification": {
      "color": "#d3d3d3"
    }
  }
}
```

This structure allows for targeted notifications with platform-specific settings and customization.


h3(#apns-headers). APNs Headers

To enable background notifications on iOS, you must include specific headers in the notification. You can specify these headers by adding an @apns-headers@ object, which should contain all the desired APNs headers, as part of the apns object. The following example shows how a background notification would appear:

```[json]
{
  "data": {
    "foo": "bar",
    "baz": "qux"
  },
  "apns": {
    "aps": {
      "content-available": 1,
    },
    "apns-headers": {
      "apns-push-type": "background",
      "apns-priority": "5"
    }
  }
}
```

h2(#push-admin). Push Admin API

Ably's "Push Admin API":/api/realtime-sdk/push-admin is a set of functionalities designed for backend servers to manage Push notification tasks. It handles registering devices for Push notifications, managing subscriptions to specific Channels, and sending Push notifications directly to devices or users identified by a client identifier. Available through realtime or REST API, the Push Admin API integrates into server-side applications, offering precise control over the Push notification process.

Using the Push Admin API requires explicit permissions within a client's credentials:

* Access using the @push-admin@ permission grants full API access, enabling the management of registrations and subscriptions for all devices.
* Access using the @push-subscribe@ permission designates a client as a Push target device. It can manage its registration and subscriptions but requires more authority over other devices' registrations or channel subscriptions.

Each Push target device is associated with a unique deviceID and authentication credentials. Authentication can be achieved in two ways when using the Push Admin API:

* By utilizing an Ably token containing the device's deviceId.
* By employing a standard "Ably key":/auth/basic or "Token":/auth/token, with a @deviceIdentityToken@, a credential generated during registration to assert the device's identity, included in the request header.

THe sevice credential management is handled by the "Ably SDK":https://ably.com/docs/sdks, removing the need for the client application to manage device credentials unless accessing the "Push Admin API":/api/realtime-sdk/push-admin directly via HTTP.
