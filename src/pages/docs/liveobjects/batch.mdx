The Batching API in LiveObjects enables multiple updates to be grouped into a single channel message and applied atomically. It ensures that all operations in a batch either succeed together or are discarded entirely. Batching operations is essential when multiple related updates to channel objects must be applied as a single atomic unit, for example, when application logic depends on multiple objects being updated simultaneously. Without batching, if one operation succeeds while another fails, your application state could become inconsistent.

Note that this differs from ["Message batching"](/docs/messages/batch), the native Pub/Sub messages feature. The LiveObjects Batching API is a separate API specifically designed to enable you to group object operations into a single channel message, ensuring that the Ably system guarantees the atomicity of the applied changes.

Batching also provides cost efficiency: all operations in a batch are sent as one channel message and billed as such, regardless of how many individual operations are included (up to the message size limit).

## Create a batch context <a id="create"/>

To batch object operations together, use the `batch()` method on a [PathObject](/docs/liveobjects/concepts/path-object) or [Instance](/docs/liveobjects/concepts/instance). This method accepts a callback function that receives a batch context. The batch context has the same API as [Instance](/docs/liveobjects/concepts/instance), except for `batch()` itself, but all mutation methods are synchronous and queue operations instead of sending them immediately. After the callback completes, all queued operations are sent together in a single channel message.

When you call `batch()` on a PathObject, the path is resolved to its underlying instance, and the batch context operates on that resolved instance.

<Aside data-type='important'>
The batch callback function must be synchronous because the batch method sends the grouped operations as soon as the callback function completes. If you need to perform asynchronous operations (such as fetching data), do so outside the callback function before calling `batch()`.
</Aside>

### Batching with PathObject <a id="pathobject"/>

Call `batch()` on a PathObject to group operations on that path:

<Code>
```javascript
const myObject = await channel.object.get();

// Batch multiple operations on the channel object
await myObject.batch((ctx) => {
  ctx.set('foo', 'bar');
  ctx.set('baz', 42);
  ctx.remove('oldKey');
});

// Batch operations on a nested path
await myObject.get('settings').batch((ctx) => {
  ctx.set('theme', 'dark');
  ctx.set('fontSize', 14);
  ctx.set('notifications', true);
});

// Batch operations on a counter
await myObject.get('visits').batch((ctx) => {
  ctx.increment(5);
  ctx.increment(3);
  ctx.decrement(2);
});
```
</Code>

### Batching with Instance <a id="instance"/>

You can also call `batch()` on an Instance to batch operations on that specific object:

<Code>
```javascript
const settingsInstance = myObject.get('settings').instance();

if (settingsInstance) {
  await settingsInstance.batch((ctx) => {
    ctx.set('theme', 'dark');
    ctx.set('fontSize', 14);
    ctx.remove('oldSetting');
  });
}
```
</Code>

If an error occurs within the batch, all operations are discarded, preventing partial updates and ensuring atomicity.

## Create objects in a batch <a id="create-objects"/>

You can create new objects inside a batch using `LiveMap.create()` and `LiveCounter.create()`. This allows you to atomically create and assign multiple objects in a single operation:

<Code>
```javascript
const myObject = await channel.object.get();

await myObject.batch((ctx) => {
  // Create and assign multiple objects atomically
  ctx.set('user', LiveMap.create({
    name: 'Alice',
    score: LiveCounter.create(0),
    settings: LiveMap.create({
      theme: 'dark',
      notifications: true
    })
  }));

  ctx.set('visits', LiveCounter.create(0));
  ctx.set('reactions', LiveMap.create({
    likes: 0,
    hearts: 0
  }));
});
```
</Code>

Creating objects inside a batch ensures that all objects are created and assigned as a single atomic operation, preventing inconsistent intermediate states.

## Understand batch context behavior <a id="context"/>

The batch context provides the same methods as [`Instance`](/docs/liveobjects/concepts/instance), except for `batch()` itself. Inside a batch callback, mutation operations are synchronous and queue operations instead of applying them immediately:

<Code>
```javascript
await myObject.batch((ctx) => {
  // These operations are synchronous and queued
  ctx.set('foo', 'bar');
  ctx.set('baz', 42);

  // Navigate to nested paths and perform operations
  const settings = ctx.get('settings');
  settings.set('theme', 'dark');

  // Counter operations are also synchronous
  const counter = ctx.get('visits');
  counter.increment(5);
  counter.increment(3);
});

// All operations are sent as a single message after the callback completes
```
</Code>

<Aside data-type='note'>
Although the batch context provides a synchronous API, updates to objects are only applied _after_ the batch callback function has run and changes have been echoed back to the client, just like regular mutation operations.
</Aside>

### Reading values in a batch <a id="reading"/>

You can read current values inside a batch context. The values reflect the state before the batch operations are applied:

<Code>
```javascript
await myObject.batch((ctx) => {
  const currentTheme = ctx.get('settings').get('theme').value();
  console.log('Current theme:', currentTheme);

  const counter = ctx.get('visits');
  const currentValue = counter.value();

  // Queue an increment operation
  counter.increment(5);

  // Reading the value still returns the pre-batch value
  console.log('Value during batch:', counter.value()); // Same as currentValue
});
```
</Code>

## When to use batching <a id="use-cases"/>

Usually, you don't need to use batching for objects operations. It is only useful in situations where a group of operations must be applied together to maintain consistency in application state, or when there are multiple mutation operations that you might want to apply at the same time to improve the UI experience.

### Maintaining consistency <a id="consistency"/>

Use batching when multiple related changes must be applied atomically:

<Code>
```javascript
// Transfer score between two players atomically
await myObject.get('players').batch((ctx) => {
  const player1Score = ctx.get('player1').get('score');
  const player2Score = ctx.get('player2').get('score');

  player1Score.decrement(10);
  player2Score.increment(10);
});
```
</Code>

### Improving UI performance <a id="performance"/>

Batch multiple updates to prevent excessive UI re-renders:

<Code>
```javascript
// Clear all tasks on a board in a single operation
await myObject.get('tasks').batch((ctx) => {
  for (const key of ctx.keys()) {
    ctx.remove(key);
  }
});

// Initialize multiple counters at once
await myObject.batch((ctx) => {
  ctx.set('likes', LiveCounter.create(0));
  ctx.set('hearts', LiveCounter.create(0));
  ctx.set('stars', LiveCounter.create(0));
  ctx.set('fire', LiveCounter.create(0));
});
```
</Code>

### Complex state updates <a id="complex-updates"/>

Use batching for complex nested structure updates:

<Code>
```javascript
// Set up a complete user profile in one atomic operation
await myObject.batch((ctx) => {
  ctx.set('user', LiveMap.create({
    id: 'user123',
    name: 'Alice',
    profile: LiveMap.create({
      bio: 'Software engineer',
      avatar: 'https://example.com/avatar.jpg',
      score: LiveCounter.create(100)
    }),
    settings: LiveMap.create({
      theme: 'dark',
      notifications: true,
      language: 'en'
    }),
    stats: LiveMap.create({
      posts: LiveCounter.create(0),
      followers: LiveCounter.create(0),
      following: LiveCounter.create(0)
    })
  }));
});
```
</Code>

## Cancel a batch <a id="cancel"/>

To explicitly cancel a batch before it is applied, throw an error inside the batch function. This prevents any queued operations from being applied:

<Code>
```javascript
await myObject.batch((ctx) => {
  ctx.set('foo', 'bar');

  const value = ctx.get('someKey').value();
  if (value === undefined) {
    // Cancel the entire batch if a required value is missing
    throw new Error('Cannot proceed: required value is missing');
  }

  ctx.set('baz', 42);
});
```
</Code>

## Understand batch constraints <a id="constraints"/>

### Cannot be used outside callback <a id="closed"/>

The batch context object cannot be used outside the callback function. Attempting to do so results in an error:

<Code>
```javascript
let context;
await myObject.batch((ctx) => {
  context = ctx;
  ctx.set('foo', 'bar');
});

// Calling any methods outside the batch callback will throw an error
try {
  context.set('baz', 42);
} catch (error) {
  console.error('Error: Batch is closed');
}
```
</Code>

## Use batching with subscriptions <a id="subscriptions"/>

When a batch operation is applied, subscribers receive a single notification for the entire batch. The notification includes all the changes made in the batch:

<Code>
```javascript
// Subscribe to the channel object
myObject.subscribe(({ object, message }) => {
  console.log('Object updated');
  if (message) {
    console.log('Operation:', message.operation.action);
  }
});

// Perform a batch operation
await myObject.batch((ctx) => {
  ctx.set('foo', 'bar');
  ctx.set('baz', 42);
  ctx.set('qux', 'hello');
});

// The subscriber receives a single notification for all three operations
```
</Code>
