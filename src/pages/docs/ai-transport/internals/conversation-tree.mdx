AI Transport maintains conversations as a tree of `RunNode`s keyed by `runId`. Each Run holds a per-Run codec projection folded from every event published under that `run-id`; the View extracts a flat `TMessage[]` from each Run's projection along the visible branch.

A conversation with one edit and one regenerate looks like:

```text
R1 (root)
  user M1: "Plan a trip to Lisbon"
  R2 ── parentRunId R1, parentCodecMessageId M1
    assistant M2: "Here's a 3-day itinerary..."
    R3 ── parentRunId R2, regeneratesCodecMessageId M2
      assistant M2': "Here's an alternative..."   ◄── regenerate sibling of R2 (anchored at M2)
    R4 ── parentRunId R2, parentCodecMessageId M2
      user M3: "Make it 5 days"
      R5 ── parentRunId R4
        assistant M4: "5-day itinerary..."
    R6 ── parentRunId R2, parentCodecMessageId M2, forkOf R4
      user M3': "Focus on food"                   ◄── edit sibling of R4 (anchored at M3)
      R7 ── parentRunId R6
        assistant M4': "Food-focused itinerary..."
```

Two kinds of sibling groups surface:

- Edit forks: R4 and R6 share `parentRunId R2` and are chained via `forkOf` on R6. The edit happened at the user prompt M3; R6 publishes a replacement user message and a fresh assistant response.
- Regenerate groups: R2 and R3 are anchored at the assistant `codec-message-id` M2. R2 owns M2; R3's `regeneratesCodecMessageId` points at M2. The View materialises one of them in the M2 slot.

Branch selection is **message-anchored** (per AITRFC-014): the branch decision lives on a `codec-message-id`, not on a `runId`. The View exposes `branchSelection(codecMessageId)` and `selectSibling(codecMessageId, index)` so a UI renders navigation arrows next to a bubble without knowing which Run owns it.

## Why a tree, not a list <a id="why-a-tree-not-a-list"/>

In a linear chat, a single list works. As soon as a user edits a message or regenerates a response, the list breaks down. Either you discard the original content (destructive) or you need a structure that can hold multiple versions.

The tree solves this. An edit creates a sibling Run rooted at the same parent message; the View can render either branch by selecting at the fork. A regenerate creates a sibling Run anchored at the assistant message being regenerated; the View materialises one of them in the bubble. Nothing is lost.

## RunNode <a id="runnode"/>

Each Run is a `RunNode<TProjection>` in the tree. The fields the Tree maintains:

| Field | Description |
| --- | --- |
| `runId` | Primary key. |
| `parentRunId` | The runId of the immediately preceding Run on this branch. Resolved from the first observed message's `parent` header via the `codec-message-id → run-id` index. |
| `parentCodecMessageId` | The `codec-message-id` this Run is rooted at. The message-level anchor. Distinct from `parentRunId` because branch filtering needs the message position, not the Run. |
| `forkOf` | The runId of the Run this Run replaces (edit fork). Resolved from the wire's `fork-of` header. |
| `regeneratesCodecMessageId` | The `codec-message-id` of the assistant message this Run regenerates. Verbatim from the wire's `msg-regenerate`. Not resolved to a runId because the anchor is a message position. |
| `clientId` | The Ably `clientId` of the client that started the Run, from `run-client-id`. |
| `status` | `'active'` between `ai-run-start` and `ai-run-end`; `'suspended'` after `ai-run-suspend` until `ai-run-resume`; otherwise the terminal `RunEndReason`. |
| `projection` | The codec's per-Run projection, folded from every event under this `run-id`. |
| `invocationId` | The agent-minted invocation id adopted from the most recent `ai-run-start` (or `ai-run-resume` for continuations). |
| `startSerial` / `endSerial` | Ably serials of the lifecycle events. Used for sibling ordering. |

## Coordination via Run lifecycle events <a id="coordination-via-run-start-and-run-continue"/>

The Tree does not arbitrate winners. Coordination is explicit:

- Lifecycle is paired: every Run begins with `ai-run-start` and ends with `ai-run-end`. The View filters active Runs by `status === 'active'`; without an `ai-run-end` the Run stays active.
- Continuations use a dedicated event: a continuation invocation (a tool-result follow-up, a suspended-Run resume) publishes `ai-run-resume` instead of a second `ai-run-start`. The client stamps `run-continue: 'true'` on the input that triggers the continuation; the agent reads that header to decide whether to emit `ai-run-start` (first start) or `ai-run-resume` (continuation). The Tree routes purely by `run-id`.
- Sibling order is by serial: when more than one Run lands at the same anchor, they are ordered by `startSerial` (oldest first). The default selection at a fork point is the newest sibling.

Earlier versions of the SDK described a "winner / precedence" model on the Tree. That model is gone. The current design relies on `Run.start()` to publish the bracketing events explicitly, and on the distinct `ai-run-resume` event to mark continuation Runs; no implicit precedence is needed.

## Sibling resolution <a id="sibling-resolution"/>

The Tree exposes two helpers for sibling navigation:

`tree.getSiblingRuns(runId)` returns every Run that is an alternative at `runId`'s fork point. The group is one of:

- Edit fork: Runs sharing a `parentRunId` and chained via `forkOf` (the original Run plus every edit that branched from its parent message).
- Regenerate group: the Run that owns a regenerated `codec-message-id` plus every Run whose `regeneratesCodecMessageId` points at it.
- Singleton: `[runId]` if neither applies.

`tree.getRegenerateGroup(runId)` returns the regenerate sibling group anchored at a `codec-message-id` and its ordered members, or `undefined` if there is no group.

The View consumes these helpers to compute `branchSelection(codecMessageId)`, which returns a `BranchSelection<TMessage>` bundle (`hasSiblings`, `siblings`, `index`, `selected`). The View's per-instance `selections` map records the selected sibling at each anchor; `selectSibling(codecMessageId, index)` updates it and emits an `'update'` event.

## Walking the tree <a id="walking-the-tree"/>

`tree.runs(selections)` returns the visible Run chain in chronological order, following the selected sibling at each fork point. The walk:

1. Start at the root Run (a Run with no `parentRunId`).
2. Add the current Run to the output list.
3. Look up the current Run's sibling group via `getSiblingRuns`. Select the Run indicated by `selections`, or default to the newest by `startSerial`.
4. Recurse into the selected Run's children: Runs whose `parentRunId` is the selected Run.
5. Stop when the selected Run has no children.

The View concatenates each visible Run's messages from `codec.getMessages(run.projection)`, then applies regenerate substitution: if a Run in the chain regenerates a `codec-message-id` whose owning Run is also in the chain, the regenerated Run's assistant content replaces the original in the message list. The result is a linear list of `TMessage` representing the currently selected conversation.

## Serial ordering and optimistic reconciliation <a id="serial-ordering-and-optimistic-reconciliation"/>

Messages within a Run are ordered by their Ably serial. The serial is the total order Ably assigns on the channel, consistent across all subscribers. Optimistic messages do not have a serial when first inserted; they sort after every confirmed message until the server publish lands. When the published event arrives the Tree promotes the placeholder to the real serial and re-sorts. In practice the change is invisible because the optimistic message was published moments before the confirmation.

## Querying the tree <a id="querying-the-tree"/>

The `Tree` interface exposes:

| Method | Description |
| --- | --- |
| `runs(selections?)` | The visible Run chain in chronological order. |
| `getRunNode(runId)` | Look up a Run by id. |
| `getRunByCodecMessageId(codecMessageId)` | Find the Run that owns a message via the `codec-message-id → run-id` index. |
| `getSiblingRuns(runId)` | The sibling group at the Run's fork point. |
| `hasSiblingRuns(runId)` | Whether the Run has alternatives. |
| `getRegenerateGroup(runId)` | The regenerate group containing the Run, anchored at a `codec-message-id`. |
| `on('update', ...)` | Structural changes (Run insert, delete, sort). |
| `on('ably-message', ...)` | Raw inbound channel messages. |
| `on('run', ...)` | Run lifecycle events. |
| `on('run-projection-updated', ...)` | Per-Run projection updates (used by the View to detect streaming deltas without a full walk). |

The View consumes these to drive its branch-aware projection. Most application code reaches for the View; the Tree is the lower-level surface for inspection.
