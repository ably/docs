Ably AI Transport adds a durable session layer to your Anthropic SDK application. The Anthropic SDK handles model calls and streaming - AI Transport handles reliable delivery, multi-device sync, and bidirectional control.

Ready to build? [Get started with Anthropic](/docs/ai-transport/getting-started/anthropic) in 5 minutes.

## Understand what each layer does <a id="layers"/>

The Anthropic SDK and AI Transport solve different problems. They stack cleanly.

| Anthropic SDK | Ably AI Transport |
| --- | --- |
| Messages API (create, stream) | Durable sessions (Ably channels) |
| Streaming events (`content_block_delta`) | Multi-device fan-out |
| Tool use and structured output | Reconnection and recovery |
| Token-level text deltas | Ordering and persistence |
| | Bidirectional control |
| | Active turn tracking |

The Anthropic SDK manages the conversation with the model. AI Transport manages the conversation with every device.

## See how they fit together <a id="architecture"/>

The integration follows a straightforward pattern:

1. **Client** sends a message via HTTP to your server.
2. **Server** calls the Anthropic Messages API with streaming enabled and receives a stream of events.
3. **Server** pipes each `text_delta` from the stream into an Ably channel.
4. **AI Transport** delivers tokens to every subscribed client with ordering, persistence, and recovery.

The HTTP response is immediate. The token stream is decoupled from the HTTP request and flows through Ably instead. This means a client can disconnect and reconnect mid-stream without losing tokens.

<Code>
```javascript
// Server: pipe Anthropic stream through Ably channel
const stream = anthropic.messages.stream({
  model: 'claude-sonnet-4-20250514',
  max_tokens: 1024,
  messages,
});

for await (const event of stream) {
  if (event.type === 'content_block_delta' && event.delta.type === 'text_delta') {
    await channel.publish('token', event.delta.text);
  }
}
```
</Code>

## Use the core transport <a id="core-transport"/>

There is no dedicated Anthropic codec in AI Transport yet. You work with the core transport directly, publishing messages and tokens to the Ably channel yourself.

This gives you full control over what gets published and how. You decide the message format, the event names, and the structure. The core transport handles turn lifecycle, cancellation, and multi-client sync.

<Code>
```javascript
// Create the server transport with the core encoder
const transport = createServerTransport({ channel, codec });

// Start a turn and stream
const turn = transport.newTurn({ turnId, clientId });
await turn.start();
```
</Code>

<Aside data-type='note'>
If you want a higher-level integration with automatic message type mapping and UI state management, consider using the [Vercel AI SDK](/docs/ai-transport/getting-started/vercel-ai-sdk) with an Anthropic provider. The Vercel integration has a dedicated codec that handles this automatically.
</Aside>

## Discover what this unlocks <a id="what-this-unlocks"/>

### Problems that go away <a id="problems-solved"/>

The Anthropic SDK streams over a single HTTP connection. That works for simple cases but breaks down in production. With AI Transport, these issues are resolved:

- Streams that die on disconnect. AI Transport resumes from where you left off, automatically.
- Lost context on tab switch or device change. The session persists across any surface.
- No way to tell if the model is still generating or has crashed. Active turn tracking gives you real-time status.
- Partial responses lost on mobile network drops. Ordering and persistence mean nothing is lost.

### New capabilities <a id="new-capabilities"/>

Things your app can do that were not possible with the Anthropic SDK alone:

- Share the same conversation on phone, laptop, and tablet, all in sync.
- Regenerate, edit, and navigate conversation branches. Each branch is a fork in the conversation tree.
- Cancel, interrupt, and steer the model mid-stream through bidirectional control.
- Approval gates reach the user on any device, even after reconnecting.
- Multiple requests stream simultaneously, each with independent cancel handles.
